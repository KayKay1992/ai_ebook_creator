## Building Authentication APIs

### What was built
Four API endpoints that handle everything related to user identity: signing up, logging in, viewing your profile, and updating it.

---

### How JWT works here (the key idea)
When a user signs up or logs in, the server creates a **JWT token** — a small encrypted string that contains the user's ID. The frontend stores this token and sends it with every future request. The server reads the token to know who is making the request, without needing to hit the database every single time to check credentials.

```
generateToken(id) → jwt.sign({ id }, JWT_SECRET, { expiresIn: '30d' })
```
The token expires in 30 days. It is signed with `JWT_SECRET` from the `.env` file.

---

### 1. Signup — `POST /api/auth/register`
**File:** `controller/authController.js` → `registerUser`

Steps:
1. Pull `name`, `email`, `password` from the request body.
2. If any field is missing → return 400 error.
3. Check if a user with that email already exists → if yes, return 400 "User already exists".
4. Create the user with `User.create(...)`. The password is **automatically hashed** by the pre-save hook in the User model (bcrypt, salt rounds 10) before it is stored.
5. Return a JWT token so the user is immediately logged in after registering.

---

### 2. Login — `POST /api/auth/login`
**File:** `controller/authController.js` → `loginUser`

Steps:
1. Pull `email` and `password` from the request body.
2. Find the user by email. Uses `.select('+password')` because the password field is hidden by default (`select: false` in the schema).
3. Call `user.matchPassword(password)` — the instance method on the User model that uses `bcrypt.compare()` to check the plain password against the stored hash.
4. If they match → return the user's info + a fresh JWT token.
5. If they don't match → return 400 "Invalid credentials".

---

### 3. Protect Middleware — `middleware/authMiddleware.js`
Before a private route runs, every request passes through the `protect` middleware first.

Steps:
1. Look for the `Authorization` header. It must start with `Bearer` (e.g. `Bearer eyJhbGc...`).
2. Split off the token part and verify it with `jwt.verify(token, JWT_SECRET)`.
3. If valid, decode it to get the user's `id` and attach the full user object to `req.user`.
4. Call `next()` to pass the request on to the actual route handler.
5. If the token is missing or invalid → return 401 "Not authorized".

This means any route that uses `protect` knows exactly who is logged in via `req.user`.

---

### 4. Get Profile — `GET /api/auth/profile` (Private)
**File:** `controller/authController.js` → `getProfile`

Steps:
1. The `protect` middleware already populated `req.user` with the logged-in user's ID.
2. Look up the user in the database by `req.user.id`.
3. Return their `_id`, `name`, `email`, `isPro`, and `avatar`.

No password is ever returned because of `select: false` on the schema.

---

### 5. Update Profile — `PUT /api/auth/profile` (Private)
**File:** `controller/authController.js` → `updateUserProfile`

Steps:
1. Find the user by `req.user.id`.
2. Update `name` and `email` if new values were provided (falls back to existing values if not).
3. If a new `password` was sent, set it — the pre-save hook will automatically hash it again on `user.save()`.
4. Save the user and return updated info + a fresh JWT token.

---

### Route Map — `routes/authRoute.js`

| Method | Endpoint              | Access  | Handler             |
|--------|-----------------------|---------|---------------------|
| POST   | /api/auth/register    | Public  | registerUser        |
| POST   | /api/auth/login       | Public  | loginUser           |
| GET    | /api/auth/profile     | Private | protect, getProfile |
| PUT    | /api/auth/profile     | Private | protect, updateUserProfile |

Public routes are open to anyone. Private routes require a valid JWT token in the `Authorization` header.
